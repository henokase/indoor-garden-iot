ESP32 Garden Controller

Complete Documentation and Implementation Guide

Contents

    1. Libraries and Dependencies

    The code includes several libraries to support the system:

    • WiFi.h: Connects the ESP32 to WiFi
    • PubSubClient.h: Handles MQTT communication for sending/receiving data
    • ArduinoJson.h: Enables easy JSON serialization and deserialization for data exchange
    • DHT.h: Interfaces with the DHT22 temperature and humidity sensor
    • NTPClient.h and WiFiUdp.h: Fetches the current time from an NTP server
    • Wire.h and LiquidCrystal_I2C.h: Supports I2C communication and controls the LCD

    2. Pin Definitions

    The code defines GPIO pins for connected devices:

    • DHT_PIN: For the DHT22 sensor
    • MOISTURE_SENSOR_PIN: Analog pin to measure soil moisture
    • FAN_PIN, PUMP_PIN, LIGHT_PIN, FERTILIZER_PIN: Output pins for controlling respective devices
    • I2C_SDA, I2C_SCL: Default I2C pins for LCD communication

    3. Constants and Configuration

    • WiFi and MQTT Credentials: Stores the credentials to connect to WiFi and MQTT broker
    • MQTT Topics: Specifies the topics for publishing sensor data, device states, and commands
    • Update Intervals: Configures time intervals for updating sensors, MQTT reconnections, and LCD updates
    • Offline Automation Thresholds: Defines temperature and moisture thresholds to control devices locally

    4. Device and Sensor States

    • DeviceState Struct: Holds the status (true for ON, false for OFF) and mode (automatic/manual) of devices
    • SensorData Struct: Tracks temperature, moisture levels, and the last update timestamp

    5. Initialization (setup function)

    • Serial Communication: Initializes serial for debugging
    • Pin Modes: Sets device pins as OUTPUT and moisture sensor as INPUT
    • Sensor Initialization: Initializes the DHT sensor
    • WiFi Connection: Calls setupWiFi() to connect the ESP32 to WiFi
    • MQTT Configuration: Sets up the MQTT broker and assigns a callback for incoming messages
    • NTP Client: Initializes the NTP client for time synchronization
    • LCD Initialization: Prepares the LCD for displaying messages

    6. Main Loop

    The loop() function manages the system's operations in real-time:

    • Connectivity Check: Periodically checks if WiFi and MQTT are connected
    • Sensor Updates: Reads temperature and moisture data at regular intervals
    • MQTT Operations: Processes MQTT messages and publishes data to the server
    • Offline Automation: Manages devices locally if the system is offline
    • LCD Updates: Updates the LCD with the latest temperature, moisture, and system status

    7. WiFi and MQTT Functions

    • setupWiFi(): Connects to WiFi and retries until successful
    • reconnectMQTT(): Attempts to reconnect to the MQTT broker and subscribes to the commands topic

    8. Sensor Data Management

    • updateSensorData(): Reads temperature and moisture data, applies moving average for stability
    • publishSensorData(): Sends the sensor data to the MQTT broker in JSON format

    9. Device State Management

    • publishDeviceStates(): Sends current device states to the MQTT broker
    • mqttCallback(): Processes incoming MQTT messages and updates device states

    10. Offline Automation

    If the system cannot connect to WiFi or MQTT:

    • enterOfflineMode(): Switches to offline mode and enables auto mode
    • exitOfflineMode(): Restores online operation and syncs states
    • handleOfflineAutomation(): Controls devices based on sensor readings:
      • Fan control based on temperature thresholds
      • Pump activation for low soil moisture
      • Time-based lighting control
      • Timed fertilizer dispensing

    11. LCD Display

    updateLCD(): Updates the LCD with:

    • Current temperature
    • Moisture level
    • Connection status (ON/OFF/NO)

    12. Key Highlights

    • Dual Operation Modes: Functions both online and offline
    • Real-Time Feedback: LCD provides live system status
    • Energy Efficiency: Automatic control based on conditions
    • Fault Tolerance: Robust connection handling and offline capabilities

    Potential Questions

    • Q: What happens if WiFi is never restored?
      A: The system will remain in offline mode and manage devices autonomously.
    • Q: How does it handle noisy sensor readings?
      A: The code applies a moving average to the moisture readings for stability.
    • Q: How does it ensure accurate time-based lighting control?
      A: Uses the NTP client to fetch the current time when online.
